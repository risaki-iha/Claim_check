# クレーム検知くん リアルタイム版

Slackの社内/社外チャンネルを分析してクレーム・トラブル・社内連携問題の兆候をスレッドの文脈込みで検知する。

## チャンネルID（固定）
- `通知先（即送信）`: C0ABRS7NR27（dxm_クレーム検知くん）

## 検知対象スコープ
- **対象**: チャンネル名に `社内` または `社外` が含まれるチャンネルのみ
- **除外**: `mdx_`、`dxm_` 等の部署内チャンネル、チャンネル名に「hajimari」を含むもの
- **除外**: Botメッセージ（include_bots: false）
- **解約リスク系は対象外**: 解約・予算削減・内製化は「解約リスク検知くん」の管轄

## 処理フロー

### Phase 1: キーワード検索（5 Agent 並列）

**検索範囲の決定（前回通知ベース）**:

C0ABRS7NR27（通知チャンネル）を `mcp__claude_ai_Slack__slack_read_channel` で読み、直近の `⚡ *クレーム検知くん リアルタイム` メッセージを探す。

**前回通知が見つかった場合**:
- 通知メッセージの `〜HH:MM` 部分（検索終了時刻）をパースし、そのUnixタイムスタンプを `after` とする
- `before`: 現在時刻のUnixタイムスタンプ
- これにより、前回カバー済みの範囲から現在までを検索する（実行が飛んでも自動的にカバー）

**前回通知が見つからない場合（初回実行 or 通知0件が続いた場合）**:
以下のフォールバックルールで検索範囲を決定する:

| 実行時刻 | 検索範囲 | 補足 |
|---------|---------|------|
| **10:00** | 前日の20:00 〜 当日10:00（約14時間） | 夜間〜朝の未検知分をまとめてカバー |
| **11:00〜20:00** | 実行時刻の1時間前 〜 実行時刻 | 直近1時間のみ |

- `after`: 上記の開始時刻のUnixタイムスタンプ（JST = UTC+9）
- `before`: 実行時刻のUnixタイムスタンプ

例: 前回通知が13:00まで → 15:00に実行 → 13:00〜15:00を検索（2時間分カバー）
例: 前回通知なし＋10:00実行 → 前日20:00〜当日10:00を検索

`mcp__claude_ai_Slack__slack_search_public_and_private` を使用。各検索のパラメータ:
- `include_bots: false`
- `sort: "timestamp"`
- `limit: 20`
- `include_context: false`
- `after`: 上記で決定した開始時刻のUnixタイムスタンプ
- `before`: 実行時刻のUnixタイムスタンプ

**チャンネルフィルター（必須）**: 検索結果のうち、`channel_name` に `社内` または `社外` が含まれるものだけを対象とする。それ以外（`mdx_`、`dxm_`、`hajimari` 等）は除外する。

### Phase 2: デデュプ＆コンテキスト取得

1. **重複除去**: 同じ `thread_ts` のメッセージを1件にまとめる
2. **チャンネル単位でグループ化**
3. **同一チャンネルの全スレッド確認**: キーワードでヒットしたスレッドだけでなく、同じチャンネル内で検索対象期間にアクティブな他のスレッドも `slack_read_channel` で確認し、関連する問題がないか網羅する
4. **スレッド読み取り**: `mcp__claude_ai_Slack__slack_read_thread` でスレッド全体を取得（文脈確保）
5. **投稿者属性判定**:
   - メールアドレスに `@nyle.co.jp` → 社内スタッフ
   - それ以外 → クライアント（社外）
6. チャンネルフィルターを再確認: `社内` または `社外` を含まないものは除外

### Phase 3: AI分析

各スレッドを以下の基準で分析する。

**🚫 大原則：脚色・行間読み禁止（最優先ルール）**

- `importance` と `summary` は、スレッド内に**実在する発言**だけを根拠にする
- 「ほのめかし」「行間」「言外の不満」「トーン・空気」を読み取って検知・昇格してはならない
- 原文に不満・指摘・問題の発言が無ければ**検知しない**（迷ったら `NOISE`）
- `summary` に原文に無い感情・態度（「そっけない」「不機嫌」「萎縮している」等）を書いてはならない
- **【発話の向き】** クレームの核は「クライアント→ナイルへの不満・指摘」。ナイルが先方に確認・回答・納品物を催促し、先方が遅延・未対応を詫びる/認めるだけのやり取り（＝待っているのはナイル側）は、ナイルへの不満ではないので**検知しない（NOISE）**。「すみません、まだです」等の謝罪は先方自身の遅れへの詫びでありクレームではない

**🧭 管轄境界（クレーム検知 ⇔ 解約リスク検知）**

この検知（クレーム検知くん）が扱うのは **ナイルの仕事・対応・品質・納期・社内連携への不満／指摘／ミス**。
次は **解約リスク検知くんの管轄**であり、ここでは検知しない（`NOISE`）:
- 解約・契約見直し・更新可否・予算削減・内製化・継続意思の低下が**主題**のもの

地続きケース（不満かつ解約予兆が混在）の裁定:
- 解約・契約終了の**明示がある** → 解約リスク側に寄せ、ここでは出さない（二重通知防止）
- 解約の明示が無く"不満だけ" → ここで検知する

**重要度の定義**:

| 重要度 | 基準 |
|-------|------|
| 🔴 即対応・上長報告 | クライアントから直接クレームや強い不満が来ている / サイト障害・インデックス外れなど外部影響が出た重大インシデント / 協力会社のミスがクライアントに影響している / 「上長報告が必要」「報告文書が必要」と社内で認識されている / 謝罪・再発防止策の提出が求められている |
| 🟡 要対応・要確認 | ナイルの対応・品質・納品物への**明確な不満や指摘**がクライアントから来ている / 費用・契約内容の変更がクライアントに未伝達 / 納期・締め切りのズレがありクライアントへの説明・謝罪が必要 / 社内スタッフがクライアントの**明確な不満・クレーム**を社内に報告している / 同じ指摘・催促が**実際に繰り返し明示**されている（複数回の催促発言が存在する場合のみ） / 放置するとクライアントとの関係や納品物に実害が出る社内連携問題（具体的なやり取りがあるもの・推測は不可） |
| 🔵 情報共有 | **【A. クライアント初期予兆】** 軽微な確認漏れ（すでに対応中・完結している） / 解決方向が見えているが記録として残したいもの / クライアントが**明示的に**初期段階の懸念を述べている（※トーン・空気からの推測は不可・実発言のみ） / 先方の遅延でナイルが待っている状況のうち、納期・納品物に実害が及ぶもの（ナイルへの不満は無いが記録は残す） ／ **【B. 社内の不穏】** 下表の社内パターンに該当するもの（すべて🔵止まり・明示発言ベース） |

**社内の不穏サイン（社内_チャンネル・🔵 情報共有として記録）**:

社内_チャンネルで以下の**明示的な発言**が見られる場合、🔵 情報共有として記録する（クライアントへの直接影響がなくても記録するが、🟡・🔴には昇格しない）:

| パターン | 具体例 | 条件 |
|---------|-------|-----------|
| 発言の引用+問い詰め | 「「〇〇」ってなんですか」「なぜ〇〇なんですか」 | 実際の引用・詰め**発言がある**場合のみ |
| 責任転嫁・言い訳への言及 | 「私は聞いていない」「なぜ把握できていなかったのか」 | 明示的な**発言がある**場合のみ |
| 繰り返しの指摘 | 「前回も」「また同じ」「再三」「以前も言いましたが」 | 実際に複数回の指摘**発言がある**場合のみ |

- ❌ **検知しない**：「遠回しなダメ出し」「萎縮・防御的な謝罪」など**トーン・空気からの推測**（大原則：脚色禁止に抵触）
- 社内の不穏は**🔵止まり**。複数パターンが重なっても🟡には昇格しない。

**ノイズ除外（出力しない＝NOISE）**:
- 社内スタッフ同士の進捗確認・作業報告・承認フロー
- レポート提出遅れなどクライアントに影響しない社内業務の遅延（「遅くなりました」「確認します」系）
- ポジティブな内容・感謝・通常の修正依頼
- 解約リスク系（解約・予算削減・内製化・継続意思 → 解約リスク検知くんの管轄）
- **クライアント側の事情**（担当者の出張・不在・休暇、先方社内の予算・体制・スケジュール調整）で、ナイルへの不満を伴わないもの
- **ナイル側が先方に確認・回答・対応を催促し、先方が遅延・未対応を詫びる/認めるやり取り**（ナイルへの不満・品質指摘を伴わないもの）はNOISE。ただし**その遅延がクライアント案件の納期・納品物に具体的な実害を及ぼす場合のみ 🔵 情報共有として記録**（🟡・🔴へ昇格しない）
- **通常の掲載可否・取材条件・素材確認の相談・調整**（不満を伴わない業務上のやり取り）
- **調査手法・個人情報・セキュリティ等への一般的な懸念・確認**（ナイルの対応への不満ではないもの）
- **ナイルの対応・品質への評価を含まないクライアント側事情全般**
- ※ただし情報漏洩・インシデント等の**重大事象は検知を維持**する

**チャンネル×投稿者の判定軸**:

| チャンネル | 投稿者 | 検知基準 |
|-----------|-------|---------|
| 社外_ | クライアント | ナイルへの不満・クレーム・品質指摘（明示発言ベース） |
| 社外_ | 社内スタッフ | クライアントの明確な不満を伝えている・謝罪・インシデント報告 |
| 社内_ | 社内スタッフ | クライアントへの実害が出る/出る可能性がある問題（🔴🟡） ／ 明示発言ベースの社内不穏（🔵止まり） |

分析結果は以下の構造で保持:
```json
{
  "channel_name": "社外_クライアント名",
  "channel_id": "C0ABC123",
  "thread_ts": "1234567890.000000",
  "importance": "🔴",
  "summary": "〇〇様より納期遅延について厳しいご指摘。担当者が謝罪中。",
  "action": "本日中に代替案を提示する必要あり",
  "is_noise": false
}
```

### Phase 4: 出力

通知本文は Python 側の `build_notification_text` が組み立てるため、AI の応答は本書の `output_schema` に従う JSON 配列のみで良い。以下は実際に Slack に送信される通知の体裁（参考）。

#### 検知なしの場合
通知自体スキップする（通知ノイズを抑えるため）。

#### 検知ありの場合

送信時は `<URL|テキスト>` 形式のリンクをそのまま維持する（プレーンURLに変換しない）。

```
⚡ *Slack - クレーム検知くん（検知期間：YYYY/MM/DD（曜） HH:MM〜HH:MM）*
━━━━━━━━━━━━━━━━━━━━
検知数：N件 ｜ 🔴 N件 ｜ 🟡 N件 ｜ 🔵 N件
━━━━━━━━━━━━━━━━━━━━
📋 詳細・ステータス管理: <https://docs.google.com/spreadsheets/d/1NYuYHOCUM-Uog5VySQ5OiAVkB5HE6_BmYPKpuELqKWI/edit?gid=419769240#gid=419769240|AI検知ログを確認する>


*━━ 🔴 即対応・上長報告 (N件) ━━*

社外_クライアント名
【対応メンバー】
山田太郎
【マネージャー】
<@U0XXXXXX>
【概要】
（スレッドの文脈を踏まえた1〜2行の要約）
🔗 <Slack URL|スレッドを見る>


*━━ 🟡 要対応・要確認 (N件) ━━*
...（🔴 と同形式。【マネージャー】行あり）


*━━ 🔵 情報共有 (N件) ━━*
...（🔴 と同形式。ただし【マネージャー】行はノイズ抑制のため出さない）
```

**【マネージャー】行のルール**:
- 🔴 🟡 のみ表示する（🔵 は出さない）
- マスタスプシ `1PWqW08yD6shJu5sRUxTZvf7w7K7TaJEuQcDU2QmkZXY` の「【公開用】案件ごとAM一覧」シートを参照（D列＝マネ名 / E列＝メアド / G列＝slackチャンネルID）
- マッチング順序: ①G列(slackチャンネルID)完全マッチ → ②B列(コール名)でチャンネル名部分マッチ
- メンション化は E列メアド → Slack user_id を優先、引けなければ D列マネ名 → Slack user_id をフォールバック、`<@U0XXXX>` 形式で記述
- どちらも引けなければ【マネージャー】行自体を出さない（事故メンション防止）
- テスト時はマスタスプシのD列/E列を直接書き換えて運用する（環境変数による差し替えは行わない）

#### Google Sheetsへの記録（検知ありの場合のみ）

スプレッドシート「クレーム検知くん」に追記する。
- スプレッドシートID: `1NYuYHOCUM-Uog5VySQ5OiAVkB5HE6_BmYPKpuELqKWI`
- シート名: `AI検知ログ`
- 書き込み方法: `gws sheets spreadsheets values append` で range に `AI検知ログ!A:L` を指定

**列構成（12列）**:
`検知媒体 | 検知内容 | 検知日時 | チャンネル名 | 重要度 | ステータス | 担当者 | 担当者アドレス | 概要 | メッセージリンク | スレッド要約 | 備考`

- **検知媒体**: `Slack`（固定）
- **検知内容**: `クレーム` または `社内トラブル`
- **検知日時**: スレッドのやり取りが発生した日時（`YYYY/MM/DD HH:MM` 形式）
- **重要度**: 絵文字＋テキストで記入 — `🔴 即対応・上長報告` / `🟡 要対応・要確認` / `🔵 情報共有`
- **担当者**: スレッドでメインにやり取りしている社内スタッフ名（`mcp__claude_ai_Slack__slack_read_user_profile` で取得）
- **担当者アドレス**: 担当者のメールアドレス（Slackプロフィールから取得）
- **ステータス**: 空欄（ユーザーが手入力）
- **備考**: 空欄（ユーザーが手入力）

## 実行時の注意

1. Phase 1の5 Agentは並列実行
2. Phase 2のスレッド取得も可能な限り並列化
3. チャンネルフィルターは Phase 1 の検索結果取得直後と Phase 2 の両方で確認する
4. 「社内_」チャンネルは「クライアントへの影響が出るか」だけでなく「不穏なやり取りがないか」も必ず判定してからノイズ除外する
5. **同一チャンネル内の全スレッド確認**: キーワードでヒットしたスレッド以外にも、同じチャンネルで検索対象期間にアクティブなスレッドがある場合はすべて読む
6. **チャンネルフィルター外の補完検索**: 「社内」「社外」を含まないが明らかにクライアント案件のチャンネル（例: `制作_クライアント名`、`セールス_案件名` 等）が検索結果にヒットした場合は、フィルターで機械的に除外せずに内容を確認する
7. 送信時は `<URL|テキスト>` 形式のリンクをそのまま維持すること（プレーンURLに変換しない）
8. Google Sheetsへの書き込みは検知ありの場合のみ実施
