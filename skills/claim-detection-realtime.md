# クレーム検知くん リアルタイム版

Slackの社内/社外チャンネルを分析してクレーム・トラブル・社内連携問題の兆候をスレッドの文脈込みで検知する。

## チャンネルID（固定）
- `通知先（即送信）`: C0ABRS7NR27（dxm_クレーム検知くん）

## 検知対象スコープ
- **対象**: チャンネル名に `社内` または `社外` が含まれるチャンネルのみ
- **除外**: `mdx_`、`dxm_` 等の部署内チャンネル、チャンネル名に「hajimari」を含むもの
- **除外**: Botメッセージ（include_bots: false）
- **解約リスク系は対象外**: 解約・予算削減・内製化は「解約リスク検知くん」の管轄

---

## 処理フロー

### Phase 1: キーワード検索（5 Agent 並列）

**検索範囲の決定（前回通知ベース）**:

C0ABRS7NR27（通知チャンネル）を `mcp__claude_ai_Slack__slack_read_channel` で読み、直近の `⚡ *クレーム検知くん リアルタイム` メッセージを探す。

**前回通知が見つかった場合**:
- 通知メッセージの `〜HH:MM` 部分（検索終了時刻）をパースし、そのUnixタイムスタンプを `after` とする
- `before`: 現在時刻のUnixタイムスタンプ
- これにより、前回カバー済みの範囲から現在までを検索する（実行が飛んでも自動的にカバー）

**前回通知が見つからない場合（初回実行 or 通知0件が続いた場合）**:
以下のフォールバックルールで検索範囲を決定する:

| 実行時刻 | 検索範囲 | 補足 |
|---------|---------|------|
| **10:00** | 前日の20:00 〜 当日10:00（約14時間） | 夜間〜朝の未検知分をまとめてカバー |
| **11:00〜20:00** | 実行時刻の1時間前 〜 実行時刻 | 直近1時間のみ |

- `after`: 上記の開始時刻のUnixタイムスタンプ（JST = UTC+9）
- `before`: 実行時刻のUnixタイムスタンプ

例: 前回通知が13:00まで → 15:00に実行 → 13:00〜15:00を検索（2時間分カバー）
例: 前回通知なし＋10:00実行 → 前日20:00〜当日10:00を検索

`mcp__claude_ai_Slack__slack_search_public_and_private` を使用。各検索のパラメータ:
- `include_bots: false`
- `sort: "timestamp"`
- `limit: 20`
- `include_context: false`
- `after`: 上記で決定した開始時刻のUnixタイムスタンプ
- `before`: 実行時刻のUnixタイムスタンプ

**チャンネルフィルター（必須）**: 検索結果のうち、`channel_name` に `社内` または `社外` が含まれるものだけを対象とする。それ以外（`mdx_`、`dxm_`、`hajimari` 等）は除外する。

#### Agent A（クレーム・不満・謝罪系）
キーワード: `クレーム` / `謝罪` / `申し訳` / `ご迷惑` / `不信感` / `不満` / `先方からFB` / `先方からフィードバック` / `早急に改善` / `改善必要な状況` / `急遽MTG`

#### Agent B（ミス・遅延・対応漏れ系）
キーワード: `ミス` / `遅延` / `指摘` / `不備` / `問題` / `確認不足` / `対応漏れ` / `対応遅れ` / `連携漏れ` / `連携遅れ` / `締め切り` / `捺印` / `検収ズレ` / `気づかず` / `催促` / `返事がない` / `返答がない` / `返答待ち` / `チェックバックが遅い` / `返信がこない` / `連絡がつかない` / `連絡が取れない` / `担当者不在` / `担当者が変わった` / `期日間違え` / `期日を間違え` / `日付間違え` / `セルがズレ` / `数値が違う` / `計算が違う` / `金額が違う` / `唐突` / `急すぎ` / `納期が短い` / `ノータッチ` / `着手していない` / `着手できていない` / `対応できません` / `リソース次第` / `リソースが足りない`

#### Agent C（契約・コスト・納期リスク系）
キーワード: `解約` / `契約見直し` / `予算削減` / `費用` / `納期` / `間に合わ`

#### Agent D（品質批判・齟齬・懸念系）
キーワード: `品質` / `クオリティ` / `齟齬` / `懸念` / `認識違い` / `改善` / `イマイチ` / `ふわっと` / `心象` / `やりにくい` / `意図が伝わっていない` / `ズレている` / `目的が不明確` / `よろしくない` / `工数に余裕がない` / `方針が変わった` / `状況整理` / `指示が不明確` / `伝わっていない` / `認識が合っていない` / `すり合わせ`

#### Agent E（技術インシデント・外部影響系）
キーワード: `インデックス` / `アクセスできない` / `閲覧できない` / `サイト` / `移管` / `障害` / `エラー` / `落ち`

### Phase 2: デデュプ＆コンテキスト取得

1. **重複除去**: 同じ `thread_ts` のメッセージを1件にまとめる
2. **チャンネル単位でグループ化**
3. **スレッド読み取り**: `mcp__claude_ai_Slack__slack_read_thread` でスレッド全体を取得（文脈確保）
4. **投稿者属性判定**:
   - メールアドレスに `@nyle.co.jp` → 社内スタッフ
   - それ以外 → クライアント（社外）
5. チャンネルフィルターを再確認: `社内` または `社外` を含まないものは除外

---

### Phase 3: AI分析

各スレッドを以下の重要度マトリクスで分析する。

**重要度の定義**:

| 重要度 | 基準 |
|-------|------|
| 🔴 即対応・上長報告 | クライアントから直接クレームや強い不満が来ている / サイト障害・インデックス外れなど外部影響が出た重大インシデント / 協力会社のミスがクライアントに影響している / 「上長報告が必要」「報告文書が必要」と社内で認識されている / 謝罪・再発防止策の提出が求められている |
| 🟡 要対応・要確認 | 費用・契約内容の変更がクライアントに未伝達 / 納期・締め切りのズレがありクライアントへの説明が必要 / 社内スタッフがクライアントの不満・懸念を社内に報告している / 放置するとクライアントとの関係や納品物に影響が出る社内連携問題 / クライアントが遠回しに不満を示している（「思っていたのと違う」「正直なところ」「聞いていない」など） / 期待値とのギャップをほのめかしている（「期待していた」「もう少し」など） / 同じ指摘・催促が繰り返されている（「前回も」「また」「再度」「何度も」など） / 困惑・戸惑いを表明している（「困っています」「少し気になっていた」など） |
| 🔵 情報共有 | 軽微な確認漏れ（すでに対応中・完結している） / 社内の気づき共有（問題になるかもしれない初期段階） / 解決方向が見えているが記録として残したいもの / クライアントのトーンが慎重・遠慮がちになっている（初期の予兆段階） |

**社内トラブル・不穏サイン（社内_チャンネル追加判定基準）**:

社内_チャンネルで以下のようなやり取りが見られる場合も検知対象とする（クライアントへの直接影響がなくても検知する）:

| パターン | 具体例 | 重要度目安 |
|---------|-------|-----------|
| 発言の引用+問い詰め | 「「〇〇」ってなんですか」「〇〇だと思うんですけど、なぜ〇〇なんですか」 | 🟡 |
| 遠回しなダメ出し・批判 | 「もう少し〜といいですね」「もう少し〜できると」「もう少し工夫が」 | 🔵〜🟡 |
| 責任転嫁・言い訳への言及 | 「自分でやったはずでは」「なぜ把握できていなかったのか」「私は聞いていない」 | 🟡 |
| 繰り返しの指摘 | 「前回も」「また同じ」「何度も」「再三」「以前も言いましたが」「繰り返しお伝えしている」 | 🟡 |
| 萎縮・防御的な謝罪 | 「緊張しておりすいません、気をつけます」など過度に委縮したトーンの謝罪 | 🔵 |

複数パターンが同一スレッドで重なる場合は 🟡 要対応・要確認 とする。

**ノイズ除外（出力しない）**:
- 社内スタッフ同士の進捗確認・作業報告・承認フロー
- レポート提出遅れなどクライアントに影響しない社内業務の遅延（「遅くなりました」「確認します」系）
- ポジティブな内容・感謝・通常の修正依頼
- 解約リスク系（解約・予算削減・内製化 → 解約リスク検知くんの管轄）

**チャンネル×投稿者の判定軸**:

| チャンネル | 投稿者 | 検知基準 |
|-----------|-------|---------|
| 社外_ | クライアント | 不満・クレーム・品質指摘 |
| 社外_ | 社内スタッフ | クライアント不満を伝えている・謝罪・インシデント報告 |
| 社内_ | 社内スタッフ | クライアントへの影響が出る/出る可能性がある問題 / 担当者間の不穏なやり取り（問い詰め・遠回しな批判・繰り返し指摘・萎縮した謝罪など） |

分析結果は以下の構造で保持:
```json
{
  "channel_name": "社外_クライアント名",
  "channel_id": "C0ABC123",
  "thread_ts": "1234567890.000000",
  "importance": "🔴",
  "summary": "〇〇様より納期遅延について厳しいご指摘。担当者が謝罪中。",
  "action": "本日中に代替案を提示する必要あり",
  "is_noise": false
}
```

### Phase 4: 出力

#### 検知なしの場合
`mcp__claude_ai_Slack__slack_send_message` で C0ABRS7NR27 に以下を即送信する:
```
【リアルタイムテスト中】
⚡ クレーム検知くん リアルタイム — YYYY/MM/DD HH:MM〜HH:MM
検知なし ✅
```

#### 検知ありの場合

`mcp__claude_ai_Slack__slack_send_message` で C0ABRS7NR27 に即送信する（下書きではない）。

**重要: slack_send_message では `*太字*` や `**太字**` の書式を絶対に使わないこと。プレーンテキストのみで送信する。`<URL|テキスト>` 形式のリンクはそのまま維持する。**

```
【リアルタイムテスト中】
⚡ クレーム検知くん リアルタイム — YYYY/MM/DD HH:MM〜HH:MM
検知: N件 ｜ 🔴 N件 ｜ 🟡 N件 ｜ 🔵 N件
📋 詳細・ステータス管理: <https://docs.google.com/spreadsheets/d/1NYuYHOCUM-Uog5VySQ5OiAVkB5HE6_BmYPKpuELqKWI/edit?gid=419769240#gid=419769240|AI検知ログを確認する>

🔴 即対応・上長報告 (N件)

チャンネル名
└ 概要: （スレッドの文脈を踏まえた1〜2行の要約）
└ アクション: （推奨アクション）
└ 🔗 <Slack URL|スレッドを見る>

🟡 要対応・要確認 (N件)
...

🔵 情報共有 (N件)
...
```

#### Google Sheetsへの記録（検知ありの場合のみ）

スプレッドシート「クレーム検知くん」に追記する。
- スプレッドシートID: `1NYuYHOCUM-Uog5VySQ5OiAVkB5HE6_BmYPKpuELqKWI`
- シート名: `AI検知ログ`
- 書き込み方法: `gws sheets spreadsheets values append` で range に `AI検知ログ!A:L` を指定

**列構成（12列）**:
`検知媒体 | 検知内容 | 検知日時 | チャンネル名 | 重要度 | ステータス | 担当者 | 担当者アドレス | 概要 | メッセージリンク | スレッド要約 | 備考`

- **検知媒体**: `Slack`（固定）
- **検知内容**: `クレーム` または `社内トラブル`
- **検知日時**: スレッドのやり取りが発生した日時（`YYYY/MM/DD HH:MM` 形式）
- **重要度**: 絵文字＋テキストで記入 — `🔴 即対応・上長報告` / `🟡 要対応・要確認` / `🔵 情報共有`
- **担当者**: スレッドでメインにやり取りしている社内スタッフ名（`mcp__claude_ai_Slack__slack_read_user_profile` で取得）
- **担当者アドレス**: 担当者のメールアドレス（Slackプロフィールから取得）
- **ステータス**: 空欄（ユーザーが手入力）
- **備考**: 空欄（ユーザーが手入力）

## 実行時の注意

1. Phase 1の5 Agentは並列実行
2. Phase 2のスレッド取得も可能な限り並列化
3. チャンネルフィルターは Phase 1 の検索結果取得直後と Phase 2 の両方で確認する
4. 「社内_」チャンネルは「クライアントへの影響が出るか」だけでなく「不穏なやり取りがないか」も必ず判定してからノイズ除外する
5. 検知なしの場合はSlack通知・Sheets記録ともに行わない
6. 送信時は `<URL|テキスト>` 形式のリンクをそのまま維持すること（プレーンURLに変換しない）
7. Google Sheetsへの書き込みは検知ありの場合のみ実施
